### 1. 논리적 완결성

- **장점**
  - 문제→해결→성공기준 흐름 자체는 대체로 명확함.
  - “외부 솔루션 연동 drop → 자체 구축 필요 → P0 unblock” 논리가 설득력 있게 제시됨.
  - KPI/성공기준과 기능 요구사항이 체계적으로 맞물림.

- **개선/결함**
  - **CQ 5축 평가의 “온톨로지 활용도” 축 정의 미흡**  
    - §7에서 오픈 이슈로 남겨두었으나, 5축 중 1개 축의 측정 기준이 불명확. 이 상태로는 “5축 자동 평가” 기능이 실질적으로 4축만 동작할 수 있음. 이는 핵심 기능의 완결성에 결함.
    - “온톨로지 활용도” 축이 불명확하다면, MVP에서 4축만으로도 성공으로 볼 것인지, 아니면 5축 완결이 필수인지 명확히 해야 함.
  - **80-20-80 HITL 워크플로우의 업무 실제 흐름/역할 분장이 부족**  
    - 각 단계별로 “누가”, “어떻게” 입력/결정/검증하는지 구체 시나리오 부족. 특히, 사람-자동화 인터페이스 경계가 모호함.
  - **Phase 2/백로그와 Core MVP의 경계가 흐림**  
    - 예컨대 “자동 상태 변경 정책”이 Phase 2로 빠졌으나, MVP 수준에서 최소한 “핸드오프 이후 업무처리(알림/트리거)”의 구체적 요구사항은 명확히 해야 현업 혼란이 적음.

### 2. 실행 가능성

- **일정**
  - **2~3 sprint(2~3주) 내 Core MVP 완결**:  
    - 인력(7명+3명)이 모두 풀가동한다면 가능할 수도 있으나, 실질적으로 병렬 투입이 어려운 영역(LLM 평가 프롬프트 엔지니어링, D1 스키마 설계, HITL UI, 통합 e2e 테스트)들이 있음.
    - 특히, Sonnet 4.6 기반의 CQ 자동 평가 프롬프트 설계/튜닝과, 사람 검수 프로세스의 UI/UX 설계가 병목.  
    - Dogfood 1회 PASS 기준도 “완결”로 보기엔 다소 약함(소규모 샘플).
    - **실제 소요는 3~5주(3 sprint+)가 더 현실적**.
- **인력**
  - **AX BD팀 7명+admin 3명**  
    - 실제로 LLM 프롬프트 설계/검증, UI/UX, 백엔드, QA를 모두 소화할 역량·경험이 있는지 명확하지 않음(컨설팅팀 중심이기 때문).
    - “프론트엔드” 담당이 TBD, DB review 담당 미정 등 핵심 역할이 미정.
- **외부/내부 의존**
  - 기존 F606/F607 토대 재활용은 타당하나, 실제 연동시 API/스키마 호환성이나 회귀 리스크가 있을 수 있음.

### 3. 누락된 핵심 요소

- **테스트/QA 체계 부재**  
  - e2e 테스트, UAT(최종 사용자 테스트) 등에 대한 별도 계획/요구사항이 없음.  
    → 최소한 “Dogfood 1회”만으로는 실제 프로덕션 투입 전 품질 보증이 부족함.  
- **프롬프트 관리/버전 관리 체계 미흡**  
  - LLM 기반 CQ 평가에서의 프롬프트 구조, 버전, 검증 방법론, 롤백/실험 전략 등 명시 필요.
- **실제 업무 프로세스와의 정합성 확인 부족**  
  - HITL Console UI가 실제 현업이 사용하기에 충분한지, 기존 admin-portal과의 교차 업무/권한 관리 등 시나리오 구체화 필요.
- **장애/예외 처리**  
  - LLM API 실패, 평가 점수 산출 오류, DB 마이그레이션 실패 등 예외 상황의 시나리오 미흡.
- **데이터 거버넌스/이력 관리**  
  - 평가 결과, 검수 이력의 보관/추적 정책, 데이터 삭제/정정 시나리오 누락.
- **보안/컴플라이언스 세부 정책**  
  - “사람만 CQ 작성” 등은 선언적으로 되어 있으나, 이를 시스템에서 어떻게 강제/감사할지 기술적 방안(예: AI 작성 방지 체계)은 미흡.

### 4. 리스크

- **핵심 기능 정의 미확정(온톨로지 활용도/역할 분리/핸드오프 범위 등)**  
  - 핵심 축/업무 흐름이 오픈 이슈 상태이므로, 개발 도중 scope creep·빈번한 기획변경 가능성 매우 높음.
- **LLM 평가 정확도 불확실**  
  - “80% 이상” 목표이나, 실제로 프롬프트 설계/튜닝 및 평가 기준 명확화 없이는 50%도 안 나올 수 있음. (실제 지표 < 50%일 경우 MVP 자체가 실패)
- **인력/역할 미정 및 실전 경험 부족**  
  - 컨설팅팀이 직접 구현/운영까지 맡을 경우, LLM 기반 평가/검증 시스템의 실전 경험 부족이 병목.
- **기존 시스템 회귀/호환성 리스크**  
  - D1 DB 스키마 추가, F606/F607 연동 등에서 기존 서비스 장애 가능성.  
- **일정 과소평가**  
  - 핵심 담당 미정, 오픈 이슈 다수, 프롬프트/UX/DB 등 병렬 진척 어려움.
- **테스트/QA 및 롤백 계획 부재**  
  - Dogfood 1회만으로는 실제 장애/품질 리스크를 커버하기 어렵고, 실패 시 롤백 전략이 없음.

### 5. 착수 판단

**착수 판단: Conditional**

**조건**
- 오픈 이슈(특히 “온톨로지 활용도” 5축 정의, HITL 워크플로우 내 구체 역할/권한/입력 흐름, 핸드오프 후 처리 범위)에 대해 최소한 MVP 기준의 명확한 결정이 선행되어야 함.
- “프롬프트 설계/검증”, “프론트엔드 담당”, “DB 구조 담당” 등 역할 분장이 명확해야 함.
- Dogfood 1회만으로 “운영 적합성”을 판단하지 않고, 별도의 QA/UAT 단계 및 실패 시 롤백 플랜을 추가해야 함.
- LLM 평가 정확도 목표 미달(50% 미만) 시의 대응 방안(프롬프트 구조 개선, fallback 등)을 사전 정의해야 함.

**요약**
- 논리 구조는 대체로 일관되나, 핵심 축 정의 및 프로세스 상세화가 미흡.
- 일정/인력은 낙관적으로 산정되어 있음.
- 테스트, 롤백, 예외 처리, 실제 사용자 업무 흐름에 대한 설계가 보완되어야 함.
- 오픈 이슈 해결 및 담당 역할 배정, QA/롤백 계획 추가를 조건으로 “Conditional” 착수 가능.
